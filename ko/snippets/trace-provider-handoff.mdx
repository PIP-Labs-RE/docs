# Trace Provider 데이터 감사 — 스테이징 인계 및 API 레퍼런스

이 문서는 Trace 데이터 감사 API의 단일 정보 출처(single source of truth)입니다. provider 통합자(온보딩, 인증, 스키마 형태, 필드 가이드, 예제)와 프런트엔드/백엔드 사용자(엔드포인트, 제한, 검증 규칙, 검색, 통계, scoped 그룹) 모두를 위한 자료입니다.

여기서 설명하는 API와 Trace Schema는 provider에 구애받지 않습니다(provider-agnostic) — 모든 provider는 동일한 엔드포인트, 헤더, 스키마를 통해 통합됩니다. 아래 구체적인 예제에서는 실제 운영 중인 provider인 Kled를 사용해 요청/응답 형태를 더 쉽게 따라갈 수 있도록 했습니다. Kled가 등장하는 모든 곳을 본인 provider의 식별 정보(`X-Provider` 값, source ID, URL, 정책 참조 등)로 대체하시면 됩니다.

## 접근 권한 얻기

쓰기(write) 접근은 게이팅되어 있습니다. 통합 전에 provider는 Story 팀에 연락해 온보딩을 받아야 합니다. 저희는 provider 식별 정보를 화이트리스트에 등록하고, 스테이징 API 키를 발급하며, 쓰기 요청에 사용해야 하는 `X-Provider` 값을 할당합니다. 이 과정이 완료되기 전까지는 쓰기 요청이 거부됩니다. 읽기, 검색, scoped 그룹 엔드포인트는 공개된 감사 뷰이므로 키가 필요하지 않습니다.

접근 권한을 요청하려면 Story 팀에 연락하여 온보딩 절차를 시작하시면 됩니다.

Base URL:

```text
https://staging-api.storyprotocol.net
```

## 버저닝

현재 V1 버전 상수:

| Version | Value | Purpose |
| --- | --- | --- |
| Trace schema | `trace-v1.0` | `initial_metadata_json` 및 `metadata_json`에 기대되는 정규화된 메타데이터 형태입니다. |
| Event hash | `event-hash-v1` | Story의 내부 감사 이벤트 해시 envelope 버전입니다. |
| Event hash canonicalization | `json-canonical-v1` | JSON은 해시 계산 전에 정규화되므로 객체 키 순서가 event hash에 영향을 주지 않습니다. |
| Search index schema | `index-v1` | 검색 인덱스 파티션 키에 임베드되어 인덱스 행에 저장되는 버전입니다. |
| Search shard schema | `shard-v1` | 샤딩된 인덱스 파티션 키에 임베드되어 인덱스 행에 저장되는 버전입니다. |

이 버전들은 영속 행(durable row)과 인덱스 행에 함께 저장됩니다. 향후 스키마/해시/샤드 변경이 발생할 때는 기존 이력을 그 자리에서 변경하기보다는 전환 기간 동안 dual-write 또는 dual-read 방식을 사용해야 합니다.

## API 흐름

```text
Provider client
  -> story-api webhook batch endpoint
  -> asynchronous Story processing
  -> story-api read/search endpoints
```

`records:batch` 응답에는 항상 항목별 상태가 포함됩니다. `202 Accepted`는 최소 하나의 항목이 비동기 처리를 위해 수락되었고 충돌한 항목이 없음을 의미합니다. `200 OK`는 모든 항목이 중복(duplicate)이어서 새로운 비동기 작업이 큐에 등록되지 않았음을 의미합니다. `409 Conflict`는 최소 하나의 항목이 충돌했음을 의미하며, 동일한 응답 내 `accepted` 상태의 항목들은 그대로 큐에 등록됩니다. 클라이언트는 항목 상태를 확인하고, 일시적 요청 실패는 동일한 payload와 동일한 `X-Batch-Id`로 재시도해야 합니다.

## 인증 및 Provider 스코프

모든 쓰기 요청에는 다음이 포함되어야 합니다:

```text
X-API-Key: <provider staging key>
X-Provider: kled
X-Batch-Id: <stable batch id>
```

쓰기 테스트에는 제공된 스테이징 API 키를 사용하세요. API 키는 사용자의 provider와 연결되어 있어야 하며, 쓰기 요청 시 `X-Provider`는 해당 provider와 일치해야 합니다. 읽기/검색/scoped-그룹 엔드포인트는 공개 감사 뷰입니다.

백로그 ingestion의 경우 `X-Ingestion-Source: backlog`를 추가하세요. 일반 라이브 ingestion에서는 헤더를 생략하세요. 명시적인 `live` 값은 허용되지 않습니다.

읽기 API는 전역 Story `data_id`로 키가 지정됩니다. provider는 응답 필드로 반환되며 선택적 필터나 검색 가능한 필드로 사용할 수 있습니다.

## 현재 Provider / Trace 정렬

provider의 초안 공개 payload는 현재 Trace Schema v1.0 워킹 형태로 깔끔하게 매핑됩니다. provider는 아래 표시된 정규화된 Trace 필드들을 전송해야 하며, provider 고유의 세부 정보가 손실되지 않도록 전체 provider 공개 payload를 `provider_payload` 아래에 포함해야 합니다.

다음 필드들은 현재 Trace v1.0 워킹 형태의 일부입니다:

- `contributor.kyc_country`: 국가 수준의 KYC 관할권 신호입니다. 국가만 포함하며, 주소나 GPS에서 파생된 국가는 포함하지 않습니다.
- `contributor.consent.tos_*` 및 `contributor.consent.privacy_policy_*`: 기여자가 해당 레코드에 대해 수락한 정확한 정책 버전과 해시입니다.
- `attestation.signed_at_utc` 및 `attestation.key_url`: 최상위 `attestation` 블록에 대한 검증 메타데이터입니다.
- `file.behavior`: PII가 아닌 캡처/업로드 동작 신호입니다.
- `file_specific.base.motion`: 미디어 유형 전반에서 공유되는 모션 신호입니다.
- `app.legal_entity`: 앱/플랫폼 이름과 함께 제공되는 법적 거래상대방 정보입니다.

provider 수준의 활성 ToS 및 Privacy Policy는 각 레코드 payload 내부가 아니라 provider 정책 엔드포인트를 통해 설정됩니다. Trace는 활성 정책 버전, 해시, URI를 저장하므로, 레코드 및 통계가 수락된 정책 참조를 provider의 현재 정책 문서와 연결할 수 있습니다.

다음 필드들은 payload에 저장되지만 현재 스테이징 구현에서는 인덱싱되지 않습니다:

- `app.legal_entity`
- `file.behavior.*`
- `file_specific.base.motion.*`
- `file.hashes.dhash64`
- `file.hashes.ahash64`
- `file.hashes.keyframe_phashes`
- `file.content_md5` / `file.hashes.md5`
- `app.platform_name`

현재의 정확 일치(exact-match) 인덱스는 Trace 프런트엔드와 감사 플로우에서 필요한 필드들을 포함합니다. 더 풍부한 provider 필드들은 여전히 저장되므로, 나중에 provider에게 과거 데이터를 다시 보내달라고 요청하지 않고도 노출하거나 인덱싱할 수 있습니다.

현재 스테이징 동작:

- 메타데이터 업데이트 상한은 스테이징에서 `data_id`당 `100`으로 변경되었습니다. 같은 provider 측 리비전의 일부인 여러 mutable 필드 변경은 하나의 메타데이터 업데이트와 하나의 `seq`로 합칠 수 있습니다.
- 읽기/검색/scoped-그룹 API는 현재 스테이징 빌드에서 공개 감사 뷰입니다. 엔터프라이즈 전용이거나 민감한 필드는 공개 Trace payload에 넣지 마세요. 향후 provider 전용 또는 파트너 전용 읽기 계층이 필요하다면, 별도의 API/제품 결정이 되어야 합니다.
- 검색 가능한 유일한 해시 유형은 canonical content SHA-256입니다. `phash64`, `dhash64`, `ahash64`, `keyframe_phashes`는 유용하다면 여전히 전송해야 하지만, 현재로서는 저장만 됩니다.
- 권장 `contributor.tax_status` 값은 `submitted`, `not_submitted`, `not_applicable`, `unknown`입니다. provider는 `tax_form_on_file=true`를 `submitted`로, `false`를 `not_submitted`로 매핑할 수 있습니다.
- 권장 `contributor.account_verification_status` 값은 `verified`, `pending`, `failed`, `unverified`입니다.
- attestation 서명은 스테이징에서 선택 사항입니다. 프로덕션 검증에서는 provider가 `payload_hash`, `signature`, `key_id`, `key_url`, `signed_at_utc`를 전송해야 합니다.
- `source_record_id`에는 provider의 안정적인 공개 미디어 ID, 예를 들어 `kmf_...`가 포함되어야 합니다. 스테이징은 주변 공백을 제거하고, 대소문자를 유지하며, 제어 문자를 거부하고, 최대 512바이트의 source ID를 수락합니다.

## Trace Schema v1.0

Story는 provider의 원본 payload를 최상위 Story 계약으로 다루지 않고, provider 정규화된 trace 메타데이터 객체를 저장합니다. provider의 전체 원본 공개 payload는 `provider_payload` 아래에 보존되어야 합니다.

provider는 표준화된 Trace Schema 필드를 직접 채워야 합니다. Story는 또한 provider의 전체 원본 공개 payload를 `provider_payload` 아래에 보존하지만, 정규화된 필드들이 다른 provider와 프런트엔드에서 사용해야 하는 이식 가능한 Trace 계약입니다.

`initial_metadata_json`과 `metadata_json`에서 `schema_version: trace-v1.0`을 사용하세요. Story는 내부 event hash를 계산하기 전에 메타데이터 JSON을 정규화하므로, 객체 키 순서는 idempotency나 충돌 감지에 영향을 주지 않습니다.

provider는 쓰기 payload에 트랜잭션 해시를 보낼 필요가 없습니다. `tx_hash`는 Story가 소유하며, 등록, 메타데이터 업데이트, 검색 결과 행에 대한 읽기 응답에서 반환됩니다. 브로드캐스트 후 Story가 채우기 전까지는 빈 문자열로 반환됩니다.

초기 등록과 메타데이터 업데이트에는 하나의 canonical content
hash가 포함되어야 합니다. 허용되는 필드는 `asset.hash`, `content_hash`, `file.content_sha256`,
또는 `file.hashes.sha256`이며, 모두 `sha256:<64-lowercase-hex>` 형식으로 정규화됩니다. 둘 이상의
alias가 존재할 경우, 동일한 해시를 나타내야 합니다.

### 필드 가이드

- `source_record_id`: provider가 소유한 안정적인 공개 미디어 ID. provider의 `kmf_...` 값은 여기에 속합니다. 스테이징은 최대 512바이트까지 허용합니다.
- `contributor.anon_id`: provider가 소유한 공개 익명화 기여자 ID.
- `contributor.kyc_status`: 권장 값은 `verified`, `pending`, `failed`, `unverified`입니다.
- `contributor.kyc_country`: 가능한 경우 KYC로부터 얻은 ISO 3166-1 alpha-2 국가 코드. 국가만 포함하며, 주소나 GPS에서 파생된 국가는 포함하지 않습니다.
- `contributor.tax_status`: 권장 값은 `submitted`, `not_submitted`, `not_applicable`, `unknown`입니다. provider의 `tax_form_on_file`에 대해 `true`를 `submitted`로, `false`를 `not_submitted`로 매핑하세요.
- `contributor.account_verification_status`: 권장 값은 `verified`, `pending`, `failed`, `unverified`입니다.
- `contributor.consent.tos_*` 및 `contributor.consent.privacy_policy_*`: 이 레코드에 대해 수락된 정책 버전, 해시, URI입니다.
- provider 활성 정책은 `PUT /webhook/v1/data-audit/provider-policy`를 통해 별도로 설정됩니다.
- `attestation.signature`: 스테이징에서는 선택 사항입니다. 프로덕션 검증에서는 provider가 `payload_hash`, `signature`, `key_id`, `key_url`, `signed_at_utc`를 전송해야 합니다.
- `file.behavior`: PII가 아닌 업로드/캡처 동작 신호입니다.
- `file_specific.base.motion`: 여러 미디어 종류에 적용될 수 있는 공유 모션 신호입니다.

## 쓰기 API

### 활성 provider 정책 설정

provider가 새로운 현행 Terms of Service 또는 Privacy Policy를 게시할 때 이 엔드포인트를 사용하세요. Trace는 활성 버전, SHA-256 해시, URI를 저장하므로, 프런트엔드가 정책 문서로 링크할 수 있고 통계는 레코드 수준에서 수락된 정책 참조를 provider의 현재 정책과 비교할 수 있습니다.

정책 문서 행은 감사/가시성 목적으로만 저장됩니다. 직접 인덱싱되거나 집계되지는 않습니다. 레코드 payload는 여전히 `contributor.consent.tos_*` 및 `contributor.consent.privacy_policy_*`를 포함해야 합니다. `/stats`, scoped 그룹 요약, 정책 해시 검색이 사용하는 것은 바로 이 레코드 수준의 수락된 정책 참조입니다.

```http
PUT /webhook/v1/data-audit/provider-policy
Content-Type: application/json
X-API-Key: <provider staging key>
X-Provider: kled
```

```json
{
  "tos": {
    "version": "2026-06-01",
    "hash": "sha256:<64-hex-policy-hash>",
    "uri": "https://kled.ai/terms/2026-06-01",
    "effective_at": "2026-06-01T00:00:00Z"
  },
  "privacy_policy": {
    "version": "2026-06-01",
    "hash": "sha256:<64-hex-policy-hash>",
```

```http
Content-Type: application/json
X-API-Key: <provider staging key>
X-Provider: kled
X-Batch-Id: kled-records-000001
X-Ingestion-Source: backlog
```

성공 응답:

- `202 Accepted`: 하나 이상의 레코드가 새로 수락되어 큐에 등록되었습니다.
- `200 OK`: 요청은 처리되었지만, 모든 항목이 `duplicate`였기 때문에 큐에 등록된 레코드가 없습니다.
- `409 Conflict`: 하나 이상의 항목이 `conflict`였습니다. 동일한 응답 내 `accepted` 상태의 항목들은 그대로 큐에 등록되었습니다.

```json
{
  "request_id": "story-request-uuid",
  "provider": "kled",
  "batch_id": "kled-records-000001",
  "format": "json",
  "kind": "records",
  "records": 1,
  "accepted": 1,
  "duplicates": 0,
  "conflicts": 0,
  "messages": 1,
  "items": [
    {
      "source_record_id": "kmf_8a9c2e7d4b1f0e23",
      "data_id": "story-generated-uuid",
      "status": "accepted"
    }
  ]
}
```

반환된 `data_id`는 향후 메타데이터 업데이트와 읽기를 위한 Story ID입니다. 동일한 `X-Provider` + `source_record_id`를 다시 보내면 동일한 `data_id`가 생성됩니다.

레코드가 이미 정확히 동일한 초기 등록 payload로 영속화되어 있었다면, 항목은 `status: "duplicate"`를 반환하고 다시 큐에 등록되지 않습니다. 동일한 `source_record_id`가 다른 초기 메타데이터로 이미 영속화되어 있었다면, 항목은 `status: "conflict"`를 반환하고 큐에 등록되지 않습니다. 첫 번째 요청이 아직 큐에 있는 동안 중첩된 재시도가 발생하면 `accepted`가 반환될 수 있으며, 하위 ingestion은 여전히 idempotent입니다. 같은 배치 내 다른 유효한 항목들은 그대로 `accepted`를 반환할 수 있습니다.

호출자가 이미 Story가 할당한 UUID를 가지고 있다면, 더 하위 수준의 엔드포인트는 다음과 같습니다:

```http
POST /webhook/v1/data-audit/data-ids:batch
```

이 엔드포인트는 모든 레코드에 `data_id`를 요구하며, 권장되는 vendor 경로는 아닙니다.

### 메타데이터 업데이트 제출

이후 수정이나 mutable 메타데이터 변경에는 이 엔드포인트를 사용하세요. `seq`는 동일한 `data_id`에 대해 `1`과 `100` 사이여야 합니다.

`metadata_json`은 canonical content hash를 포함해야 하며, 부분 diff나 JSON patch가 아닌 변경 이후 전체 최신 Trace 메타데이터 상태여야 합니다. 예를 들어 KYC만 변경되더라도, provider는 여전히 현재 메타데이터 상태의 일부로 남아 있는 변경되지 않은 file, asset, app, consent, provider payload 필드들을 포함해야 합니다. 이를 통해 각 메타데이터 이벤트가 `metadata_root`에 대해 독립적으로 검증 가능해지고, Story는 provider별 병합 규칙 없이 최신 상태를 재구성할 수 있습니다.

```http
POST /webhook/v1/data-audit/metadata-updates:batch
Content-Type: application/json
X-API-Key: <provider staging key>
X-Provider: kled
X-Batch-Id: kled-metadata-000001
X-Ingestion-Source: backlog
```

요청 본문은 JSON 배열입니다:

```json
[
  {
    "data_id": "11111111-1111-4111-8111-111111111111",
    "seq": 1,
    "prev_metadata_root": "sha256:<previous-canonical-trace-schema-v1-json>",
    "metadata_root": "sha256:<new-canonical-trace-schema-v1-json>",
    "metadata_json": {
      "schema_version": "trace-v1.0",
      "asset": {
        "hash": "sha256:<64-hex>"
      },
      "contributor": {
        "anon_id": "kup_123",
        "kyc_status": "unverified",
        "consent": {
          "tos_version": "2026-06-01",
          "tos_hash": "sha256:<64-hex-policy-hash>",
          "tos_uri": "https://kled.ai/terms/2026-06-01",
          "privacy_policy_version": "2026-05-20",
          "privacy_policy_hash": "sha256:<64-hex-policy-hash>",
          "privacy_policy_uri": "https://kled.ai/privacy/2026-05-20"
        }
      },
      "app": {
        "platform_name": "kled.ai"
      },
      "provider_payload": {
        "media_id_public": "kmf_8a9c2e7d4b1f0e23",
        "reason": "kyc_status_changed"
      }
    },
    "occurred_at": "2026-05-13T00:00:01Z"
  }
]
```

`metadata_root`는 업데이트된 canonical 전체 Trace Schema v1.0 메타데이터 JSON에 대한 provider의 결정론적이고 0이 아닌 해시여야 합니다. `prev_metadata_root` 역시 0이 아닌 root여야 합니다. 현재 스테이징 API에서는 Story가 이 값들을 제출된 그대로 저장하며, `metadata_json`에 대한 해시 검증은 아직 강제되지 않습니다.

### 선택적 백로그 파일 엔드포인트

일반적인 통합 경로는 `/records:batch`에서 `application/json`을 사용하는 것입니다. 백로그 도구용으로는 다음과 같은 하위 수준 라우트 변형들도 존재하지만, 데이터 ID 파일 라우트는 모든 레코드에 `data_id`를 요구합니다.

```text
POST /webhook/v1/data-audit/data-ids:batch-ndjson
POST /webhook/v1/data-audit/metadata-updates:batch-ndjson
POST /webhook/v1/data-audit/data-ids:batch-csv
POST /webhook/v1/data-audit/metadata-updates:batch-csv
POST /webhook/v1/data-audit/data-ids:batch-txt
POST /webhook/v1/data-audit/metadata-updates:batch-txt
```

NDJSON에는 다음이 필요합니다:

```text
Content-Type: application/x-ndjson
Content-Encoding: gzip
```

압축 해제된 각 줄은 위에 표시된 동일한 필드를 가진 하나의 JSON 레코드입니다.

CSV에는 `Content-Type: text/csv`가 필요합니다.

Data ID CSV 컬럼:

```text
data_id,source_record_id,initial_metadata_root,initial_metadata_json,occurred_at
```

메타데이터 업데이트 CSV 컬럼:

```text
data_id,seq,prev_metadata_root,metadata_root,metadata_json,occurred_at
```

`initial_metadata_json`과 `metadata_json`은 따옴표로 묶인 CSV 필드 내에서 유효한 JSON이어야 하며 하나의 canonical content hash를 포함해야 합니다.

TXT에는 `Content-Type: text/plain`이 필요하며, 줄 단위 JSON, JSON 배열, 또는 동일한 CSV 컬럼을 사용하는 헤더 구분 콤마/탭 텍스트를 허용합니다.

## 읽기 API

읽기 엔드포인트는 공개 감사 뷰입니다. `provider`는 선택 사항이며 좁히는 필터로 작동합니다.

읽기 모델:

- `GET /data-ids/{data_id}`는 등록 프로필과 최신 원본 메타데이터 이벤트를 반환합니다. 최신 메타데이터 이벤트가 반드시 diff인 것은 아닙니다. 현재 저장된 가장 높은 시퀀스에 대해 제출된 정확한 업데이트 payload입니다.
- `GET /data-ids/{data_id}/metadatas`는 `seq: 0`의 등록과 이후 메타데이터 업데이트를 포함한 전체 append-only 메타데이터 이력을 반환합니다.
- 검색, asset receipt 조회, scoped 그룹 요약은 이 이벤트들에서 파생된 Story의 정규화된 latest-state projection을 사용합니다. 이 projection은 MIME 타입, 미디어 카테고리, KYC 상태, TOS/privacy 버전, lifecycle 상태, `tx_hash` 같은 필드를 구동합니다.

MIME 타입, 미디어 카테고리, KYC 상태, 국가/지역, TOS, Privacy Policy 버전과 같은 광범위한 분포 카운트는 `/stats`를 사용하세요. UI가 가장 최신 레코드를 필요로 할 때는 `/recent` 또는 `/feed`를 사용하세요.
`source_record_id` 검색은 정확 일치이며 대소문자를 구분합니다.

기타 예제:


### content hash로 Asset receipt 조회

```http
GET /api/v1/data-audit/assets/sha256:aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa
```

허용되는 해시 형식은 `sha256:<64-hex>`, 일반 `<64-hex>`, `0x<64-hex>`, `0x1220<64-hex>`입니다. 응답은 canonical `sha256:<64-lowercase-hex>` asset hash와 일치하는 모든 receipt 행을 반환합니다.

### Scoped 그룹

Scoped 그룹은 Trace 프런트엔드나 파트너 리뷰어가 검토 집합에 대해 공개 스냅샷을 생성할 수 있게 해줍니다. provider 리뷰 워크플로의 경우, lab은 Story가 생성한 `data_id`나 content hash가 아니라 provider의 `source_record_id` 값을 사용해야 합니다. Trace는 결정론적 `data_id`를 계산하고, 제출된 모든 레코드가 존재하는지 검증하며, 전체 집합이 유효한 경우에만 그룹을 생성합니다.

source record ID로부터 생성:

```http
POST /api/v1/data-audit/scoped-groups
Content-Type: application/json
Idempotency-Key: kled-review-000001
```

```json
{
  "title": "Kled review set",
  "description": "Kled source record IDs supplied to a lab",
  "provider": "kled",
  "source_record_ids": ["kmf_1", "kmf_2"]
}
```

붙여넣은 source record ID로부터 생성:

```json
{
  "title": "Kled pasted review set",
  "provider": "kled",
  "source_record_ids_text": "kmf_1\nkmf_2\nkmf_3"
}
```

Source-record 그룹은 전부 아니면 전무(all-or-nothing) 방식입니다. 중복된 `provider + source_record_id` 값은 `400`을 반환하고, 누락된 레코드는 `404`를, 프로필/source 불일치는 `409`를, asset projection 없이 등록된 레코드는 `422`를 반환합니다.

더 큰 source-record CSV/TXT 입력의 경우, 먼저 presigned 업로드 URL을 요청하세요:

```http
POST /api/v1/data-audit/scoped-groups/uploads
Content-Type: application/json
```

```json
{ "format": "csv" }
```

지원되는 업로드 형식은 `csv`와 `txt`입니다.

단일 컬럼 CSV는 `source_record_id`를 사용하며 생성 요청에 `provider`가 필요합니다:

```csv
source_record_id
kmf_1
kmf_2
```

여러 provider가 섞인 CSV는 두 컬럼을 모두 포함하며 최상위 `provider`가 필요하지 않습니다:

```csv
provider,source_record_id
kled,kmf_1
oto,oto_1
```

TXT 업로드는 줄바꿈으로 구분된 source record ID이며 생성 요청에 `provider`가 필요합니다:

```text
kmf_1
kmf_2
```

반환된 `Content-Type` 헤더와 함께 반환된 `upload_url`에 파일 바이트를 업로드한 다음, 다음으로 그룹을 생성하세요:

```json
{
  "title": "Kled uploaded review set",
  "description": "CSV uploaded through presigned S3",
  "provider": "kled",
  "upload_id": "up_<uuid>.csv"
}
```

업로드된 CSV에 `provider` 컬럼이 있을 때만 `provider`를 생략하세요.

Scoped 그룹 읽기:

```http
GET /api/v1/data-audit/scoped-groups/{group_id}
GET /api/v1/data-audit/scoped-groups/{group_id}/items?limit=100
GET /api/v1/data-audit/scoped-groups/{group_id}/items?limit=100&cursor=<next_cursor>
GET /api/v1/data-audit/scoped-groups/{group_id}/export.csv
```

`GET /scoped-groups/{group_id}`는 그룹 상태를 반환하며, 완료되면 `summary`에 집계 지표를 함께 반환합니다:

```json
{
  "profile": {
    "group_id": "sg_...",
    "title": "Kled review set",
    "description": "Kled source record IDs supplied to a lab",
    "manifest_kind": "source_record",
    "status": "complete",
    "submitted_items": 2,
    "unique_items": 2,
    "computed_at": "2026-06-04T15:30:00Z"
  },
  "summary": {
    "records_in_set": 2,
    "submitted_items": 2,
    "unique_items": 2,
    "matched_items": 2,
    "missing_items": 0,
    "matched_receipts": 2,
    "kyc_verified_percent": 50,
    "distinct_media_categories": ["image", "video"],
    "distinct_tos_versions": ["2026-05"],
    "total_size_bytes": 7340032,
    "average_size_bytes": 3670016,
    "source_distribution": { "kled": 2 },
    "media_category_coverage": { "image": 1, "video": 1 },
    "mime_distribution": { "image/jpeg": 1, "video/mp4": 1 },
```

`/items`는 제출된 각 `source_record_id`마다 한 행을 반환하며, 해결된 `data_id`, `asset_hash`, receipt 요약을 포함합니다. 커서 기반 페이지네이션이며 더 많은 행이 있을 때 `next_cursor`를 반환합니다. CSV export는 `input_type,provider,source_record_id,data_id,asset_hash,status,...`로 시작합니다.

## 제한 및 재시도 동작

현재 스테이징 제한:

| Limit | Value |
| --- | --- |
| 최대 요청 본문 크기 | `25 MiB` |
| 최대 SQS 메시지 청크 | `240 KiB` |
| 최대 SQS 배치 payload | `256 KiB` |
| 최대 직렬화 레코드 크기 | `350 KiB` |
| 데이터 ID당 최대 메타데이터 업데이트 수 | `100` |
| 최대 인라인 scoped 그룹 본문 | `5 MiB` |
| 최대 인라인 scoped 그룹 해시 수 | `10,000` |
| 최대 인라인 scoped 그룹 source 레코드 수 | `10,000` |
| 최대 source_record_id 길이 | `512 bytes` |

재시도 가이드:

- `502`, `503`, `504`, 네트워크 타임아웃, `429`는 지수 백오프와 지터를 적용해 재시도하세요.
- 검증/인증 `4xx`는 요청이 수정되기 전까지 재시도하지 마세요.
- 재시도 시 `data_id`, 요청 본문, `X-Batch-Id`를 일정하게 유지하세요.
- `X-Ingestion-Source: backlog`는 백로그 작업에만 사용하고, 라이브 작업에서는 생략하세요.
- 쓰기 경로는 동일한 `data_id`, event key, event hash에 대해 idempotent입니다.
- 동일한 `data_id`와 event key가 서로 다른 메타데이터로 재시도되면, 해당 레코드는 충돌로 간주되어 거부됩니다.

## 검증 규칙

| Rule | Behavior |
| --- | --- |
| 쓰기 엔드포인트에 `X-Provider` 누락 | 요청이 거부됩니다. |
| 잘못된 provider 이름 | 요청이 거부됩니다. |
| 구성된 allowlist 외부의 provider | 쓰기 요청이 거부됩니다. |
| `WEBHOOK` 키가 `X-Provider`와 일치하지 않음 | 쓰기 요청이 거부됩니다. |
| 쓰기 엔드포인트에 `X-Batch-Id` 누락 | 요청이 거부됩니다. |
| `X-Ingestion-Source`가 `backlog` 이외의 값으로 존재 | 요청이 거부됩니다. |
| `/records:batch`에 `source_record_id` 누락 | 요청이 거부됩니다. |
| 하나의 `/records:batch` 요청 내 중복된 `source_record_id` | 요청이 거부됩니다. |
| 동일한 초기 등록 payload를 가진 기존 `/records:batch` 항목 | 항목이 `status: "duplicate"`를 반환합니다. |
| 다른 초기 등록 payload를 가진 기존 `/records:batch` 항목 | 항목이 `status: "conflict"`를 반환합니다. |
| 하위 수준 데이터 ID 및 메타데이터 업데이트 라우트에서 UUID가 아닌 `data_id` | 요청이 거부됩니다. |
| `occurred_at` 누락 | 요청이 거부됩니다. |
| 잘못된 `occurred_at` 타임스탬프 | 요청이 거부됩니다. |
| `1`에서 `100` 범위를 벗어난 메타데이터 `seq` | 요청이 거부됩니다. |
| 누락되거나 형식이 잘못되거나 0인 필수 메타데이터 root 필드 | 요청이 거부됩니다. |

## 참고 사항

- 전달은 최소 한 번(at-least-once)이므로 중복 제출이 발생할 수 있습니다.
- 동일한 이벤트의 중복 제출은 idempotent하게 처리됩니다.
- 동일한 `data_id`, 동일한 메타데이터 시퀀스, 다른 이벤트 콘텐츠는 충돌로 처리됩니다.
- 메타데이터 업데이트는 초기 데이터 ID 등록보다 먼저 도착할 수 있습니다.
- 감사 데이터는 영속적이며 만료되지 않습니다.
- Event hash는 정규화된 메타데이터 JSON에 event/hash/schema 버전 필드를 더해 계산됩니다.
